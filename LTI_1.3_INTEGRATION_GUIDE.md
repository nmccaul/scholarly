## 1. Core Concepts

### Entities

| Entity | Description |
|--------|-------------|
| **Platform** | The LMS — in our case, Canvas |
| **Tool** | Our external application (Scholarly) |
| **Deployment** | A specific integration instance between Platform and Tool; generates an immutable `deployment_id` (max 255 ASCII chars) |
| **Resource Link** | A reference to a specific tool resource stored in Canvas; has a platform-unique `resource_link_id` |
| **Context** | A course or group — organizes resource links |

### Key IDs to Track

- `deployment_id` — identifies the platform-tool integration; used to scope service tokens and match launches
- `resource_link.id` — identifies a specific LTI link within a context; platform-unique
- `sub` (user ID) — permanent, unique user identifier (max 255 ASCII chars); stable across launches
- `client_id` — the OAuth 2.0 identifier issued to our tool during Canvas developer key creation

## 2. Security Model

LTI 1.3 replaced OAuth 1.0a (used in LTI 1.1) with **OpenID Connect + OAuth 2.0**.

### Key Principles

- All communication over **HTTPS / TLS 1.2+**. SSL is explicitly prohibited.
- Platform-originating launch messages are **OpenID Connect ID tokens** (signed JWTs).
- Tool-to-platform service calls use **OAuth 2.0 Bearer tokens** (JWT Client Credentials grant).
- **Asymmetric cryptography only** — RS256 (RSA + SHA-256) by default.
- Keys are distributed via **JWKS endpoints** (JSON Web Key Sets) to allow rotation without re-registration.

### JWT Signing

```
Header: { "typ": "JWT", "alg": "RS256" }
```

- Platform signs launch JWTs with its **private key**; tool verifies with the platform's **public key** (from JWKS endpoint).
- Tool signs service-request JWTs with its **private key**; platform verifies with the tool's **public key**.

### CSRF & Replay Prevention

| Mechanism | Where Used |
|-----------|-----------|
| `nonce` | Required in every ID token and tool-originated JWT; tool must reject previously-seen nonces |
| `state` | Generated by tool in Step 2; validated by tool in Step 4 to prevent CSRF |
| `exp` claim | Token expiry — validate current time is before `exp` |
| `iat` claim | Token issue time — optionally validate within an acceptable window |

## 3. Launch Flow (Step-by-Step)

```
User          Canvas (Platform)              Our Tool
 |                   |                           |
 |  clicks LTI link  |                           |
 |------------------>|                           |
 |                   |  1. OIDC Login Init       |
 |                   |  POST /oidc_initiation_url|
 |                   |-------------------------->|
 |                   |                           |  2. Tool builds auth request
 |  <-- 302 redirect to Canvas OIDC auth endpoint (browser-only, GET/POST) ---|
 |                   |                           |
 |  3. Browser hits Canvas OIDC authorize URL    |
 |------------------>|                           |
 |                   |  validates client_id &    |
 |                   |  redirect_uri             |
 |                   |                           |
 |  4. Canvas POSTs signed JWT (id_token) to tool's redirect_uri              |
 |<------------------|-------------------------->|
 |                   |                           |  5. Tool validates JWT
 |                   |                           |  6. Tool grants access
 |<-- renders resource -------------------------|
```

### Step 1 — Login Initiation (Canvas → Tool)

Canvas sends a GET or POST to the tool's `oidc_initiation_url`:

| Parameter | Notes |
|-----------|-------|
| `iss` | Canvas issuer (environment-dependent) |
| `login_hint` | Opaque; must be echoed back in Step 2 |
| `target_link_uri` | Intended final resource URL |
| `client_id` | Tool's OAuth2 client ID |
| `deployment_id` | Tool deployment identifier |
| `canvas_region` | AWS region (Canvas-specific) |
| `canvas_environment` | `production`, `beta`, or `test` |
| `lti_message_hint` | Optional; metadata about pending LTI message |
| `lti_storage_target` | Present when platform storage (cookie-free) is supported |

### Step 2 — Authentication Request (Tool → Canvas)

Tool redirects the **browser** (not server-to-server) to Canvas's OIDC authorization endpoint:

```
GET https://sso.canvaslms.com/api/lti/authorize_redirect
  ?scope=openid
  &response_type=id_token
  &client_id=<CLIENT_ID>
  &redirect_uri=<REGISTERED_REDIRECT_URI>
  &login_hint=<VALUE_FROM_STEP_1>
  &state=<RANDOM_CSRF_TOKEN>
  &response_mode=form_post
  &nonce=<UNIQUE_VALUE>
  &prompt=none
```

**All six parameters are required.** Generate a fresh `state` and `nonce` for every launch; store them server-side (or via LTI Platform Storage) to validate in Step 4.

### Step 3 — Canvas Validates & Signs JWT

Canvas validates `client_id` and `redirect_uri`, then constructs a signed ID token (JWT) containing the full LTI payload.

### Step 4 — Authentication Response (Canvas → Tool)

Canvas POSTs to the tool's `redirect_uri` (form POST):

```
POST <redirect_uri>
  id_token=<SIGNED_JWT>
  &state=<ECHO_OF_STATE>
```

Tool must:
1. Verify `state` matches what was stored in Step 2.
2. Fetch Canvas's public JWKS and verify JWT signature.
3. Validate `iss`, `aud`, `exp`, `nonce`, `deployment_id`.
4. Parse LTI claims and serve the resource.

### Step 5 — Resource Display

Tool redirects to `target_link_uri` and renders the resource (typically in an iframe).

## 4. JWT Structure & Claims

### Standard OIDC Claims (always present)

| Claim | Type | Description |
|-------|------|-------------|
| `iss` | string | Platform issuer URL |
| `sub` | string | Unique user ID (max 255 chars); absent for anonymous launches |
| `aud` | string | Tool's `client_id` |
| `exp` | number | Expiry (Unix epoch seconds) |
| `iat` | number | Issued-at (Unix epoch seconds) |
| `nonce` | string | Replay protection; must not be reused |
| `azp` | string | Authorized party (when multiple audiences exist) |

### Required LTI Claims

| Claim URI | Description |
|-----------|-------------|
| `https://purl.imsglobal.org/spec/lti/claim/message_type` | `LtiResourceLinkRequest` |
| `https://purl.imsglobal.org/spec/lti/claim/version` | `1.3.0` |
| `https://purl.imsglobal.org/spec/lti/claim/deployment_id` | Platform-tool deployment ID |
| `https://purl.imsglobal.org/spec/lti/claim/target_link_uri` | Must match login initiation value |
| `https://purl.imsglobal.org/spec/lti/claim/resource_link` | Object: `{ id, title?, description? }` |
| `https://purl.imsglobal.org/spec/lti/claim/roles` | Array of role URIs |

### Important Optional LTI Claims

| Claim URI | Description |
|-----------|-------------|
| `https://purl.imsglobal.org/spec/lti/claim/context` | `{ id, type[], label, title }` — the course |
| `https://purl.imsglobal.org/spec/lti/claim/tool_platform` | `{ guid, name, url, product_family_code, version, ... }` |
| `https://purl.imsglobal.org/spec/lti/claim/launch_presentation` | `{ document_target, height, width, return_url, locale }` |
| `https://purl.imsglobal.org/spec/lti/claim/lis` | `{ person_sourcedid, course_offering_sourcedid, ... }` |
| `https://purl.imsglobal.org/spec/lti/claim/custom` | Key-value map of custom variables |
| `https://purl.imsglobal.org/spec/lti/claim/role_scope_mentor` | Array of student IDs accessible to a mentor |

### AGS Service Claim (present when grading is available)

```json
"https://purl.imsglobal.org/spec/lti-ags/claim/endpoint": {
  "scope": ["https://purl.imsglobal.org/spec/lti-ags/scope/lineitem", ...],
  "lineitems": "https://<canvas>/api/lti/courses/<id>/line_items",
  "lineitem": "https://<canvas>/api/lti/courses/<id>/line_items/<id>"
}
```

### NRPS Service Claim (present when roster is available)

```json
"https://purl.imsglobal.org/spec/lti-nrps/claim/namesroleservice": {
  "context_memberships_url": "https://<canvas>/api/lti/courses/<id>/names_and_roles",
  "service_versions": ["2.0"]
}
```

### Anonymous Launches

If `sub` is absent, treat the user as anonymous. The `roles` claim should still be included (may be empty).

## 5. Tool Registration in Canvas

### Developer Key Setup (Canvas Admin)

1. Go to **Admin → Developer Keys → + Developer Key → LTI Key**
2. Provide or paste the tool's JSON configuration (see Section 6)
3. Enable the key (toggle to ON)
4. Note the numeric **Client ID** — this is our `client_id`
5. Deploy the key to the account or sub-account

### Minimum Registration Data Exchanged Out-of-Band

| Item | Direction |
|------|-----------|
| Tool's public key or JWKS URL | Tool → Canvas |
| `oidc_initiation_url` | Tool → Canvas |
| `target_link_uri`(s) / `redirect_uri`(s) | Tool → Canvas |
| Canvas issuer URL | Canvas → Tool |
| Canvas OIDC auth endpoint | Canvas → Tool |
| Canvas token endpoint | Canvas → Tool |
| Canvas JWKS endpoint | Canvas → Tool |
| `client_id` (issued on registration) | Canvas → Tool |

## 7. Deep Linking (LTI-DL 2.0)

Used when Canvas needs to pick content from our tool (e.g., adding an assignment, embedding content in the Rich Content Editor).

### Flow

1. Canvas sends an `LtiDeepLinkingRequest` JWT to our tool.
2. Tool presents a content picker UI.
3. User selects content.
4. Tool constructs an `LtiDeepLinkingResponse` JWT and auto-submits it via form POST back to Canvas's `deep_link_return_url`.

### Deep Linking Request — Required Claims

| Claim | Value |
|-------|-------|
| `message_type` | `LtiDeepLinkingRequest` |
| `version` | `1.3.0` |
| `deployment_id` | Platform-tool deployment ID |
| `https://purl.imsglobal.org/spec/lti-dl/claim/deep_linking_settings` | Object containing `deep_link_return_url`, `accept_types[]`, `accept_presentation_document_targets[]` |

### Deep Linking Response — Required Claims

| Claim | Value |
|-------|-------|
| `message_type` | `LtiDeepLinkingResponse` |
| `version` | `1.3.0` |
| `deployment_id` | Must match request |
| `aud` | Platform's issuer URL |
| `iss` | Tool's `client_id` |
| `content_items` | Array of selected items (may be empty) |
| `data` | Echo of `data` from the request (if present) |

### Content Item Types

| Type | Use Case |
|------|----------|
| `ltiResourceLink` | LTI launch link (supports grading via `lineItem`) |
| `link` | External URL with optional window/iframe rendering |
| `html` | Raw HTML fragment (Canvas sanitizes) |
| `image` | Direct image embed |
| `file` | File transfer URL |

### Example: Returning an LTI Resource Link with Grading

```json
{
  "type": "ltiResourceLink",
  "title": "Scholarly Assignment",
  "url": "https://<our-domain>/lti/launch/assignment",
  "lineItem": {
    "scoreMaximum": 100,
    "label": "Scholarly Assignment",
    "resourceId": "assignment-123",
    "tag": "scholarly"
  },
  "available": {
    "startDateTime": "2026-01-01T00:00:00Z",
    "endDateTime": "2026-12-31T23:59:59Z"
  },
  "submission": {
    "endDateTime": "2026-12-31T23:59:59Z"
  }
}
```

### Response Delivery

Tool must auto-submit a form POST:

```html
<form method="POST" action="<deep_link_return_url>">
  <input type="hidden" name="JWT" value="<SIGNED_RESPONSE_JWT>" />
</form>
<script>document.forms[0].submit();</script>
```

## 8. Assignment & Grade Services (AGS 2.0)

Allows our tool to read and write grades in Canvas's gradebook without screen scraping.

### Service Endpoints (from JWT claim)

| Endpoint | Description |
|----------|-------------|
| `lineitems` URL | Container — list/create line items for the context |
| `lineitem` URL | Specific line item (present only for single-lineitem launches) |

### OAuth 2.0 Scopes

| Scope | Permitted Operations |
|-------|---------------------|
| `https://purl.imsglobal.org/spec/lti-ags/scope/lineitem` | GET, POST, PUT, DELETE line items |
| `https://purl.imsglobal.org/spec/lti-ags/scope/lineitem.readonly` | GET line items only |
| `https://purl.imsglobal.org/spec/lti-ags/scope/result.readonly` | GET results (read grades) |
| `https://purl.imsglobal.org/spec/lti-ags/scope/score` | POST scores (submit grades) |

### Getting an Access Token

```http
POST <canvas_token_endpoint>
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer
&client_assertion=<TOOL_SIGNED_JWT>
&scope=https://purl.imsglobal.org/spec/lti-ags/scope/score
```

The `client_assertion` JWT must contain:
- `iss` = tool's `client_id`
- `sub` = tool's `client_id`
- `aud` = Canvas token endpoint URL
- `iat` = now
- `exp` = now + 300 (5 minutes)
- `jti` = unique token ID

Token response:
```json
{
  "access_token": "...",
  "token_type": "bearer",
  "expires_in": 3600,
  "scope": "..."
}
```

### Line Item Structure

```json
{
  "id": "https://<canvas>/api/lti/courses/<id>/line_items/<id>",
  "scoreMaximum": 100,
  "label": "Scholarly Essay Grade",
  "resourceId": "scholarly-essay-1",
  "tag": "scholarly",
  "startDateTime": "2026-01-01T00:00:00Z",
  "endDateTime": "2026-12-31T23:59:59Z",
  "gradesReleased": true
}
```

### Submitting a Score

```http
POST <lineitem_url>/scores
Authorization: Bearer <ACCESS_TOKEN>
Content-Type: application/vnd.ims.lis.v1.score+json

{
  "userId": "<sub from launch JWT>",
  "scoreGiven": 85,
  "scoreMaximum": 100,
  "activityProgress": "Completed",
  "gradingProgress": "FullyGraded",
  "timestamp": "2026-05-05T12:00:00Z",
  "comment": "Great work!"
}
```

**`activityProgress` values:** `Initialized`, `Started`, `InProgress`, `Submitted`, `Completed`

**`gradingProgress` values:** `FullyGraded`, `Pending`, `PendingManual`, `Failed`

### Retrieving Results

```http
GET <lineitem_url>/results
Authorization: Bearer <ACCESS_TOKEN>
Accept: application/vnd.ims.lis.v2.result+json
```

### Line Item Filtering

```http
GET <lineitems_url>?resource_link_id=<id>&resource_id=<id>&tag=scholarly&limit=50
```

### Media Types

| Media Type | Used For |
|-----------|---------|
| `application/vnd.ims.lis.v2.lineitem+json` | Single line item |
| `application/vnd.ims.lis.v2.lineitemcontainer+json` | Multiple line items |
| `application/vnd.ims.lis.v1.score+json` | Score submission |
| `application/vnd.ims.lis.v2.result+json` | Grade result |

> **Important:** Always re-fetch the current line item state before PUT updates — Canvas manages some fields; preserve them.

## 9. Names & Role Provisioning Services (NRPS 2.0)

Lets our tool fetch the full course roster with roles.

### Endpoint (from JWT claim)

`context_memberships_url` in the `namesroleservice` claim.



```json
{
  "id": "<context_memberships_url>",
  "context": {
    "id": "<course_id>",
    "label": "CS101",
    "title": "Intro to Computer Science"
  },

    {
      "user_id": "<matches sub in LTI launch>",

      "status": "Active",
      "name": "Jane Student",
      "given_name": "Jane",
      "family_name": "Student",
      "email": "jane@university.edu",
      "picture": "https://...",
      "lis_person_sourcedid": "sis-id-123"
    }
  ]

```

### Query Parameters

| Parameter | Description |
|-----------|-------------|
| `role` | Filter by role (e.g., `role=Learner`) |

## 12. Cookie-Free Launch Support

Safari and some browsers block third-party cookies inside iframes. Canvas provides a postMessage-based storage API to handle this.

### Detection

Canvas includes `lti_storage_target` parameter in the login initiation request when it supports this feature.

### Using Platform Storage

Instead of setting a session cookie, use `postMessage` to store and retrieve `state`:

**Store state (during Step 2):**
```javascript
window.parent.postMessage({
  subject: 'lti.put_data',
  key: 'state',
  value: stateValue
}, 'https://sso.canvaslms.com');
```

**Retrieve state (during Step 4 validation):**
```javascript
window.parent.postMessage({
  subject: 'lti.get_data',
  key: 'state'
}, 'https://sso.canvaslms.com');
```

The target origin for postMessage is the OIDC auth domain (e.g., `https://sso.canvaslms.com`), provided via `lti_storage_target`.

Canvas confirms support by responding to a `lti.capabilities` postMessage with `lti.get_data` and `lti.put_data` in the capabilities list.

## 13. Conformance & Security Checklist

### Tool Endpoints Required

- [ ] `GET/POST /lti/login` — OIDC initiation URL
- [ ] `GET /lti/jwks` — Tool's public JWKS endpoint
- [ ] `POST /lti/launch` — OIDC redirect URI (handles `id_token` + `state`)
- [ ] `POST /lti/launch/deep_link` — Deep linking response target (if using deep linking)

### JWT Validation Checklist (every launch)

- [ ] Fetch Canvas's JWKS and verify JWT signature (RS256)
- [ ] `iss` matches the expected Canvas issuer for the environment
- [ ] `aud` contains our `client_id`
- [ ] `exp` is in the future
- [ ] `nonce` has not been seen before (store seen nonces with TTL = token lifetime)
- [ ] `state` matches what was stored in Step 2
- [ ] `deployment_id` is recognized
- [ ] `target_link_uri` matches a known endpoint

### Service Call Security

- [ ] Always use HTTPS
- [ ] Request access tokens with the minimum required scopes
- [ ] Treat `access_token` as opaque
- [ ] Re-fetch service URLs from the latest launch JWT (Canvas may change them)
- [ ] Handle token expiry and request renewal before `expires_in` elapses

### General Security

- [ ] All endpoints behind TLS 1.2+
- [ ] JWKS endpoint serves RSA public keys only (never private keys)
- [ ] State parameter is cryptographically random (min 128 bits)
- [ ] Nonce is unique per launch (UUID or similar)
- [ ] Unknown claims are silently ignored (forward-compatibility requirement)
